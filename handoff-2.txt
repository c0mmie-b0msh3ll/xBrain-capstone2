1. Current state

TF1 AI Ops hiện là event-driven, compute-first triage service.

High-level flow:

Incident seed/event
-> bounded evidence lookup
-> normalized triage context
-> deterministic RCA/scoring
-> optional AgentCore grounded investigation
-> catalog-gated recommended actions
-> report JSON/API
-> Slack payload + Jira ticket_payload

Quan trọng:

- Shared handoff cho CDO phải transport-neutral.
- Không đưa implementation detail riêng của bất kỳ CDO team nào vào docs/contracts chung.
- AI không gọi trực tiếp vào customer application để lấy log/metrics.
- Extra data phải đến từ CDO-hosted observability/evidence path.
- AI/LLM không có quyền write/remediate/restart/rollback/scale.

4. Safety boundaries

AgentCore is not allowed to execute remediation directly.

AgentCore can only ask TF1 for extra evidence through allowlisted read-only tool calls:

- get_metrics
- get_logs
- get_recent_deploys
- get_ownership
- detect_metric_anomalies
- detect_log_anomalies
- infer_topology
- infer_causal_hints
- rank_rca_candidates

TF1 validates:

- tenant_id
- environment
- service
- bounded time window
- max tool calls
- log limit
- max window minutes

Final recommended actions come from `action_catalog.py`. LLM/AgentCore cannot invent action IDs or executable remediation types.

8. Current Slack/Jira state

Slack:

- One-way notification/dry-run through Incoming Webhook.
- No two-way Slack bot yet.
- No slash commands, buttons, thread Q&A, or approval flow yet.

Jira:

- TF1 generates `ticket_payload`.
- TF1 does not create Jira issues directly yet.
- Current demo flow: TF1 Slack notify/report -> owner reviews -> Jira manually or integration consumes `ticket_payload`.

Assignment policy:

- Known service routes to owner team/project/channel.
- Unknown service routes to default triage queue.
- Noisy/low-confidence routes to triage queue.
- Critical/high-confidence routes to owner queue.
- No direct personal assignment unless Jira accountId mapping exists.

11. Jira publisher future work

Preferred approach:

Backend or integration layer creates Jira from `ticket_payload`. Slack should not be the source of record for ticket creation.

Reason:

- Backend has audit trail.
- Backend has RCA evidence.
- Backend has confidence/status/classification.
- Backend can enforce idempotency by incident_id/audit_id.
- Backend can attach report URL.
